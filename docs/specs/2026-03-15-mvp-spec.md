# VibeView MVP Spec — Claude Code Support

**Date:** 2026-03-15
**Scope:** Local CLI tool + web server for viewing Claude Code sessions

---

## Overview

VibeView is a local CLI tool that serves a web UI for browsing and viewing Claude Code conversation sessions. It watches Claude Code's data directories, discovers sessions, and streams updates in real time.

The MVP supports **Claude Code only**. Future versions will add Codex and Gemini CLI.

---

```
vibeview (Go binary)
```

## Claude Code Data Model

Claude Code stores session data at:

- **Session index:** `~/.claude/history.jsonl` — one JSON object per line, each entry containing:
  - `sessionId` — UUID
  - `project` — absolute path to the project directory
  - `display` — the user's message text (first line / preview)
  - `timestamp` — Unix timestamp in milliseconds

- **Session files:** `~/.claude/projects/{encoded-project-path}/{sessionId}.jsonl` — one JSON object per line, each representing a message/event.

### Project path encoding

The project directory path is encoded by replacing `/` with `-` (e.g., `/Users/foo/myproject` becomes `-Users-foo-myproject`).

### Message types in session JSONL

Each line is a JSON object with a `type` field:

| Type | Description |
|------|-------------|
| `user` | User message. `message.content` is a string or array of content blocks (text, `tool_result`) |
| `assistant` | Assistant message. `message.content` is an array of content blocks (text, `tool_use`, thinking) |
| `progress` | Progress events (hook execution, etc.) |
| `system` | System events (stop hooks, summaries) |
| `file-history-snapshot` | File state snapshots (can be ignored for display) |

### Common fields on message objects

- `uuid` — unique message ID
- `parentUuid` — parent message ID (for threading)
- `sessionId` — session UUID
- `timestamp` — ISO 8601 timestamp
- `cwd` — working directory at time of message
- `version` — Claude Code version
- `gitBranch` — active git branch
- `slug` — human-readable session name (on later messages)
- `isSidechain` — whether this is a side-chain message

### Assistant message content blocks

```jsonc
// Text
{ "type": "text", "text": "..." }

// Tool use
{ "type": "tool_use", "id": "toolu_...", "name": "Bash", "input": { "command": "...", "description": "..." } }

// Thinking (when present)
{ "type": "thinking", "thinking": "..." }
```

### User message content blocks (tool results)

```jsonc
// Plain text
"content": "user typed this"

// Tool results (array)
[
  { "type": "tool_result", "tool_use_id": "toolu_...", "content": "output...", "is_error": false }
]
```

### Usage / cost data

Assistant messages include `message.usage`:
```jsonc
{
  "input_tokens": 3,
  "output_tokens": 516,
  "cache_creation_input_tokens": 14,
  "cache_read_input_tokens": 44420
}
```

## CLI

```
vibeview [flags]

Flags:
  --port, -p        Port to serve on (default: 1337)
  --claude-dir      Path to Claude Code data dir (default: ~/.claude)
  --open            Open browser on start (default: true)
```

On start, the CLI:
1. Discovers all sessions from `history.jsonl`
2. Starts the HTTP server
3. Begins watching for file changes (new sessions, new messages)
4. Opens the browser (unless `--open=false`)

## API Endpoints

All endpoints are prefixed with `/api`.

### `GET /api/sessions`

Returns a list of all discovered sessions.

```jsonc
{
  "sessions": [
    {
      "id": "49fb3b61-...",
      "project": "/Users/foo/myproject",
      "display": "Initialize this project...",
      "timestamp": "2026-02-15T12:34:13.085Z",
      "messageCount": 52,
      "model": "claude-opus-4-6",       // from first assistant message
      "slug": "twinkling-bouncing-metcalfe"  // if available
    }
  ]
}
```

Supports query params:
- `?project=` — filter by project path (substring match)
- `?sort=recent` — sort by most recent (default)

### `GET /api/sessions/:id`

Returns full session detail with all messages.

```jsonc
{
  "session": {
    "id": "49fb3b61-...",
    "project": "/Users/foo/myproject",
    "messages": [
      {
        "uuid": "a7349045-...",
        "type": "user",
        "timestamp": "2026-02-15T12:34:13.085Z",
        "content": { /* parsed content */ }
      },
      {
        "uuid": "77ada3e4-...",
        "type": "assistant",
        "timestamp": "2026-02-15T12:34:15.411Z",
        "model": "claude-opus-4-6",
        "content": { /* parsed content */ },
        "usage": { /* token counts */ }
      }
    ]
  }
}
```

### `GET /api/sessions/:id/stream`

SSE endpoint for real-time updates. Emits events as new lines are appended to the session JSONL file.

```
event: message
data: { "uuid": "...", "type": "assistant", ... }

event: message
data: { "uuid": "...", "type": "user", ... }
```

Also emits a `ping` event every 30s to keep the connection alive.

### `GET /api/health`

Returns `{ "status": "ok" }`.

## Frontend Pages

### Session List (`/`)

- Shows all sessions grouped by project
- Each session card shows: project name, slug/display text, timestamp, model, message count
- Search/filter by project name
- Click to open a session
- Auto-updates when new sessions appear (SWR polling or SSE)

### Session View (`/session/:id`)

- Chat-style layout, read-only
- Message types rendered differently:
  - **User messages** — right-aligned or distinct style, plain text
  - **Assistant messages** — left-aligned, with markdown rendering
  - **Tool calls** — collapsible blocks showing tool name, input, and result
  - **Thinking blocks** — collapsible, visually distinct (e.g., muted/italic)
  - **File edits** — syntax-highlighted diff view (for Write/Edit tool calls paired with their results)
  - **System/progress** — small, muted inline indicators
- **Cost sidebar or header** — shows total tokens (input/output/cache) for the session
- **Live updates** — connects to SSE endpoint, appends new messages as they arrive
- Scroll-to-bottom behavior for live sessions

---

## Real-Time Updates

### Backend

- Use `fsnotify` (or similar) to watch session JSONL files for changes
- When a file is modified, read new lines appended since last known position
- Push new messages to connected SSE clients
- Watch `history.jsonl` for new session entries

### Frontend

- Session list: SWR with periodic revalidation (every 5s)
- Session view: SSE connection via `EventSource`, new messages appended to local state
- Visual indicator when a session is "live" (actively receiving messages)

## Non-Goals (MVP)

- Codex or Gemini CLI support
- Authentication / multi-user
- Remote deployment
- Writing or modifying sessions
- Conversation forking / branching visualization
- Export functionality
- Mobile-optimized layout
